15class MobileApplicationClient(Client): 

16 

17 """A public client utilizing the implicit code grant workflow. 

18 

19 A user-agent-based application is a public client in which the 

20 client code is downloaded from a web server and executes within a 

21 user-agent (e.g. web browser) on the device used by the resource 

22 owner. Protocol data and credentials are easily accessible (and 

23 often visible) to the resource owner. Since such applications 

24 reside within the user-agent, they can make seamless use of the 

25 user-agent capabilities when requesting authorization. 

26 

27 The implicit grant type is used to obtain access tokens (it does not 

28 support the issuance of refresh tokens) and is optimized for public 

29 clients known to operate a particular redirection URI. These clients 

30 are typically implemented in a browser using a scripting language 

31 such as JavaScript. 

32 

33 As a redirection-based flow, the client must be capable of 

34 interacting with the resource owner's user-agent (typically a web 

35 browser) and capable of receiving incoming requests (via redirection) 

36 from the authorization server. 

37 

38 Unlike the authorization code grant type in which the client makes 

39 separate requests for authorization and access token, the client 

40 receives the access token as the result of the authorization request. 

41 

42 The implicit grant type does not include client authentication, and 

43 relies on the presence of the resource owner and the registration of 

44 the redirection URI. Because the access token is encoded into the 

45 redirection URI, it may be exposed to the resource owner and other 

46 applications residing on the same device. 

47 """ 

48 

49 response_type = 'token'

51 def prepare_request_uri(self, uri, redirect_uri=None, scope=None, 

52 state=None, **kwargs): 

53 """Prepare the implicit grant request URI. 

54 

55 The client constructs the request URI by adding the following 

56 parameters to the query component of the authorization endpoint URI 

57 using the "application/x-www-form-urlencoded" format, per `Appendix B`_: 

58 

59 :param redirect_uri: OPTIONAL. The redirect URI must be an absolute URI 

60 and it should have been registerd with the OAuth 

61 provider prior to use. As described in `Section 3.1.2`_. 

62 

63 :param scope: OPTIONAL. The scope of the access request as described by 

64 Section 3.3`_. These may be any string but are commonly 

65 URIs or various categories such as ``videos`` or ``documents``. 

66 

67 :param state: RECOMMENDED. An opaque value used by the client to maintain 

68 state between the request and callback. The authorization 

69 server includes this value when redirecting the user-agent back 

70 to the client. The parameter SHOULD be used for preventing 

71 cross-site request forgery as described in `Section 10.12`_. 

72 

73 :param kwargs: Extra arguments to include in the request URI. 

74 

75 In addition to supplied parameters, OAuthLib will append the ``client_id`` 

76 that was provided in the constructor as well as the mandatory ``response_type`` 

77 argument, set to ``token``:: 

78 

79 >>> from oauthlib.oauth2 import MobileApplicationClient 

80 >>> client = MobileApplicationClient('your_id') 

81 >>> client.prepare_request_uri('https://example.com') 

82 'https://example.com?client_id=your_id&response_type=token' 

83 >>> client.prepare_request_uri('https://example.com', redirect_uri='https://a.b/callback') 

84 'https://example.com?client_id=your_id&response_type=token&redirect_uri=https%3A%2F%2Fa.b%2Fcallback' 

85 >>> client.prepare_request_uri('https://example.com', scope=['profile', 'pictures']) 

86 'https://example.com?client_id=your_id&response_type=token&scope=profile+pictures' 

87 >>> client.prepare_request_uri('https://example.com', foo='bar') 

88 'https://example.com?client_id=your_id&response_type=token&foo=bar' 

89 

90 .. _`Appendix B`: https://tools.ietf.org/html/rfc6749#appendix-B 

91 .. _`Section 2.2`: https://tools.ietf.org/html/rfc6749#section-2.2 

92 .. _`Section 3.1.2`: https://tools.ietf.org/html/rfc6749#section-3.1.2 

93 .. _`Section 3.3`: https://tools.ietf.org/html/rfc6749#section-3.3 

94 .. _`Section 10.12`: https://tools.ietf.org/html/rfc6749#section-10.12 

95 """ 

96 return prepare_grant_uri(uri, self.client_id, self.response_type, 

97 redirect_uri=redirect_uri, state=state, scope=scope, **kwargs)

99 def parse_request_uri_response(self, uri, state=None, scope=None): 

100 """Parse the response URI fragment. 

101 

102 If the resource owner grants the access request, the authorization 

103 server issues an access token and delivers it to the client by adding 

104 the following parameters to the fragment component of the redirection 

105 URI using the "application/x-www-form-urlencoded" format: 

106 

107 :param uri: The callback URI that resulted from the user being redirected 

108 back from the provider to you, the client. 

109 :param state: The state provided in the authorization request. 

110 :param scope: The scopes provided in the authorization request. 

111 :return: Dictionary of token parameters. 

112 :raises: OAuth2Error if response is invalid. 

113 

114 A successful response should always contain 

115 

116 **access_token** 

117 The access token issued by the authorization server. Often 

118 a random string. 

119 

120 **token_type** 

121 The type of the token issued as described in `Section 7.1`_. 

122 Commonly ``Bearer``. 

123 

124 **state** 

125 If you provided the state parameter in the authorization phase, then 

126 the provider is required to include that exact state value in the 

127 response. 

128 

129 While it is not mandated it is recommended that the provider include 

130 

131 **expires_in** 

132 The lifetime in seconds of the access token. For 

133 example, the value "3600" denotes that the access token will 

134 expire in one hour from the time the response was generated. 

135 If omitted, the authorization server SHOULD provide the 

136 expiration time via other means or document the default value. 

137 

138 **scope** 

139 Providers may supply this in all responses but are required to only 

140 if it has changed since the authorization request. 

141
